# 🛡️ المعمارية التقنية لوكيل الويب في الإنتاج (Production Web Agent Stack)

تم تكييف تقنيات وكيل الويب ليتوافق تماماً مع **الهيكل الجديد الموجه للإنتاج (Production)** الذي يضم (Fast Filter Drivers -> Envoy Cluster -> HA SOC).

---

## 1. الطبقة 0: تعريف الـ Kernel Drivers (فلترة السرعة القصوى)
لن يكون الوكيل معتمداً فقط على بيئة واحدة. 
*   **بيئة Linux:** سنستمر باستخدام `XDP/eBPF`. الأكواد مكتوبة بـ C و Rust لفلترة الحزم في 100 نانوثانية بناءً على تقييم الـ IPs و Rate Limiting.
*   **بيئة Windows:** سيتم كتابة مشغل (Driver) منوع `NDIS LWF` مبرمج بـ C/C++ مع الحصول على تصريحات موثقة (Signed/WHQL). هذا يوفر فلترة في زمن 10 إلى 50 ميكروثانية.

## 2. الطبقة 1: بيئة الـ Envoy Sidecar
استبدلنا أدوات الـ Uprobes المتفرقة بـ `Envoy Proxy` يعمل كعنقود (Cluster):
*   **إنهاء التشفير العالي:** يدير Envoy مفاتيح `TLS 1.3` مع إمكانية تحديث الشهادات التشغيلية دون أوقات توقف (Certificate Rotation).
*   **Hot Restart:** تحديث قواعد الوكيل ومرشحات الجدار الناري لن يسبب فقداناً لاتصالات الشبكة الفعالة.
*   **Circuit Breakers:** طبقة حماية ذاتية لوكيلنا. إذا استغرق الفحص الدلالي وقتاً طويلاً، سيتحول الوكيل لوضع الـ Fail-Open أو Fail-Closed (حسب إعداد المسؤول) لمنع شل الخدمة.
*   **Load Balancing (Least Request):** في حال وجود أكثر من تطبيق موازٍ، سيقوم Envoy بتوزيع الحمل بصورة موثوقة.

## 3. محركات الحماية (Thor Extensions on Envoy)
*   دمج محرك البحث الكلاسيكي (`Coraza / ModSecurity`) لحظر 90% من الهجمات من خلال `WASM`.
*   إرسال الطلبات المتجاوزة إلى محرك تحليل دلالي (Thor Engine) يعمل جنباً إلى جنب.

## 4. الطبقة 2: تكامل SOC عبر أنظمة التراسل (Kafka / RabbitMQ)
بدل إرسال البيانات مباشرةً عبر gRPC بسيط، سيقوم Envoy ومحرك Thor بدفع (Push) سجلات الملخصات الذكية إلى طابور رسائل (Queue) لامركزي.
*   هذا يضمن أن الوكلاء (Sidecars) لن يتأثروا بأي عطل في الـ SOC المركزي.
