# 🚀 خطة التنفيذ الميداني: بناء وكيل الويب والشبكة (Thor Envoy Sidecar)

بناءً على اعتماد الهيكل المعماري الشامل والدقيق الذي طلبته (XDP -> Envoy Sidecar -> SOC)، تم تحديث خطة العمل لتتوافق تماماً مع هذا الهيكل المتين (Enterprise-Grade). سنعمل على تنفيذ (Sidecar) ليتم نشره بجانب كل تطبيق/سيرفر.

---

## 🏗️ المرحلة 1: طبقة النواة (XDP Kernel Drops)
**الهدف:** التخلص من الهجمات الغبية بسرعة البرق قبل وصولها لـ Envoy.
*   **المهام البرمجية:**
    1.  كتابة برنامج eBPF/XDP يعترض الحزم في الطبقة 3/4.
    2.  تجهيز `eBPF Map` (LPM Trie) لتخزين الـ IPs المحظورة والـ GeoIP Blocks بشكل محلي.
    3.  برمجة خوارزمية (Token Bucket) داخل الـ XDP لمنع الاتصالات الكثيفة من IP واحد في وقت قصير (Rate Limiting).
    4.  كتابة أداة Rust (`thor-agent-net`) لتحديث جداول XDP لحظياً عند استلام أمر من SOC.

## ⚡ المرحلة 2: وكيل Envoy وقواعد Coraza 
**الهدف:** تأسيس الـ Sidecar وإنهاء التشفير والحماية الكلاسيكية (OWASP).
*   **المهام البرمجية:**
    1.  إعداد إعدادات `envoy.yaml` لتمكين (TLS Termination) واستقبال طلبات الويب.
    2.  تجميع وتفعيل محرك `Coraza WAF` داخل Envoy إما عبر (WASM) أو كإضافة مدمجة، وتشغيل قواعد `ModSecurity Core Rule Set (CRS)`.
    3.  إضافة فلاتر الشبكة (Network Filters) داخل Envoy.
*   **المخرج:** WAF Sidecar صلب ينهي التشفير ويحمي التطبيقات خلفه من 90% من ثغرات الويب المعروفة بزمن استجابة 1ms.

## 🧠 المرحلة 3: محرك Thor الذكي (Thor Engine & Semantic Analysis)
**الهدف:** كشف الهجمات المعقدة والـ Zero-day التي تخطت ModSecurity.
*   **المهام البرمجية:**
    1.  إنشاء خدمة `Rust Ext_Authz` (أو WASM plugin) تعمل جنباً إلى جنب مع Envoy. كل طلب يتجاوز Coraza، يُرسل إلى هذه الخدمة للتقييم.
    2.  دمج `ONNX Runtime` داخل محرك Thor، وتحميل نموذج ذكاء اصطناعي خفيف جداً (Small Semantic Model وزن 10-50MB) مهمته قراءة الهيكل الدلالي للطلب.
    3.  دمج أداة (API Schema Validation) بحيث يرفض المحرك أي طلب لا يتوافق هيكلياً مع (Swagger/OpenAPI) الخاص بالتطبيق.
*   **المخرج:** قدرة فائقة على فهم سياق الهجوم وليس مجرد البحث عن كلمات عبر Regex.

## 🔄 المرحلة 4: التحليل السلوكي المحلي (Local Redis)
**الهدف:** رصد السلوكيات الشاذة وسحب البيانات (Scraping / Credential Stuffing) محلياً.
*   **المهام البرمجية:**
    1.  ربط `Thor Engine` بخدمة `Redis` محلية تعمل ضمن الصلاحية ذاتها.
    2.  تتبع (State) لكل جلسة أو IP: تكرار تسجيل الدخول الخاطئ، الوصول السريع جداً لمسارات متعددة.

## 📡 المرحلة 5: التكامل الذكي مع SOC (Out-of-band telemetry)
**الهدف:** التواصل السريع والمؤمن لنقل الأحداث وتحديث القواعد.
*   **المهام البرمجية:**
    1.  برمجة عميل gRPC يقرأ ملخصات الأحداث (الشذوذ، الهجومات المستهدفة، ملخص السلوك) ويرسلها للـ SOC (Fire-and-forget لكي لا يبطئ الطلب الفعلي).
    2.  تجهيز مسار (Stream) لاستقبال القوائم السوداء الجديدة من الـ SOC وإرسالها فوراً إلى طبقة XDP (المرحلة 1).

---

### نقطة البدء الفعلية:
بناءً على طلبك اعتماد هذه الخطة حرفياً، فإن **الخطوة العملية الأولى** التي سنبدأ بها الآن هي:
1. تجهيز بيئة (XDP/eBPF) مع أداة مساعدة في Rust (`thor-agent-net`) قادرة على حظر قوائم الـ IP بسرعة البرق (المرحلة 1).
2. إعداد حاوية (Docker/Envoy) لتكون الـ Sidecar التجريبي.
